# blog-post

SEO 최적화 블로그 포스팅을 멀티 에이전트 방식으로 생성한다.
티스토리 HTML + 마크다운 두 형식으로 저장한다.

실제 콘텐츠 제작 프로세스로 동작한다:
PD가 방향을 잡고 → 기획자가 SEO 전략을 세우고 → 에이전트 토론으로 제목/구성을 확정하고 → 작가가 집필하고 → 편집장이 최종 결재한다.

## 실행 전 준비
blog-output/ 폴더가 없으면 프로젝트 루트에 생성한다.

---

## STEP 1 — [PD] 프로젝트 분석 + 글 방향 설정

모든 에이전트의 작업 기준이 되는 브리핑을 만든다.
이 단계에서 방향이 잘못 잡히면 이후 모든 작업이 틀어지므로 가장 신중하게 수행한다.

**분석 순서 (반드시 이 순서):**

1. **핵심 가치 먼저 정의** — 코드보다 먼저
   - 이 프로젝트는 무엇을 왜 만들었는가?
   - 어떤 문제를 해결하는가?
   - 독자가 이 글을 읽고 얻어가야 할 것은?

2. **프로젝트 구조 파악**
   - 디렉토리 트리 확인 (depth 2~3)
   - package.json / pubspec.yaml 등 의존성 파일
   - README가 있으면 읽기

3. **핵심 구현 선별** — 핵심 가치와 직결된 것만
   - 핵심 가치를 구현한 코드/로직
   - 독창적이거나 설명할 가치가 있는 구현
   - 10~20줄 이내 코드 스니펫 2~3개 선별

4. **차별 포인트 탐색** (선택)
   - 다른 방식 대신 이 방식을 택한 이유
   - 겪은 어려움이나 트레이드오프 (있으면 활용, 없으면 건너뜀)
   - 커밋 메시지에서 단서 확인 가능하면 확인

5. **글의 카테고리 + 타겟 독자 확정**
   - 글 유형: 튜토리얼 / 개념 설명 / 비교 분석 / 구현 가이드 중 선택
   - 타겟: 주니어 / 시니어 / 비개발자 중 선택
   - 독자가 이미 알고 있을 것 vs 설명해야 할 것

출력 형식:
```
[PD 브리핑]
■ 핵심 가치: (1문장 — 이 글 전체의 기준)
■ 글 유형:
■ 타겟 독자: (구체적으로)
■ 핵심 구현 (파일명 + 1줄 설명):
  1.
  2.
■ 차별 포인트: (있으면 기술, 없으면 "없음")
■ 기획자에게 전달할 방향:
  - 반드시 다뤄야 할 것:
  - 다루지 말아야 할 것:
```

## STEP 2 — [기획자] SEO 전략 + 콘텐츠 구조 설계

PD 브리핑을 기반으로 SEO 전략을 수립하고 글의 뼈대를 설계한다.

### 2-1. 검색 의도 분석

독자가 구글에서 이 글을 찾게 되는 상황을 구체적으로 정의한다.

- **검색 의도 유형**: Informational(정보 탐색) / How-to(방법 탐색) / Comparison(비교) 중 선택
- **독자 시나리오**: "독자는 지금 어떤 상황에서 무엇을 검색창에 치는가?" (1~2문장으로)
- **메인 키워드**: 독자가 구글 검색창에 실제로 칠 쿼리 1개
  - 기준: 이 키워드로 검색했을 때 내 글이 답이 되는가?
  - "개발기", "이야기", "후기" 같은 감성 표현 금지
- **서브 키워드**: 본문에 자연스럽게 녹일 관련 검색어 4~5개
- **LSI 키워드**: 메인 키워드와 함께 등장하는 연관 개념어 3개
  - 예: 메인="Claude Code 팀 설정" → LSI: "settings.json", "~/.claude", "slash command"

### 2-2. 경쟁 콘텐츠 분석 (추정)

이 키워드로 이미 존재할 경쟁 글을 추정하고 차별화 포인트를 찾는다.
- 예상 경쟁 콘텐츠 유형: (공식 문서 / 해외 블로그 / 없음 등)
- 우리 글의 차별점: (실제 코드 포함 / 한국어 유일 / 팀 적용 사례 등)
- Featured Snippet 狙い 가능 여부: 구글 답변 박스에 뜰 수 있는 형식인가?

### 2-3. 제목 후보 3개 (SEO 근거 포함)

각 후보마다 SEO 관점 근거를 함께 제시한다.

제목 기준:
- 메인 키워드가 앞쪽에 배치 (구글은 앞 키워드에 더 가중치)
- 40~60자 (구글 SERP에서 잘림 없이 노출)
- 검색 의도를 직접 담은 동사형 ("~하는 법", "~설정 방법", "~구현 가이드")
- 클릭베이트 완전 금지: "개발기", "이야기", "배운 것들", "N가지", "알아보겠습니다"
- 브랜드명/고유명사만 단독으로 쓰지 않기 (검색량 없음)

### 2-4. 글 구조 설계

고정 섹션 없음 — 검색 의도와 글 유형에 맞는 구조를 새로 설계한다.

- 도입부: 독자의 문제 상황을 즉시 제시 + 메인 키워드 첫 100자 이내 등장
- 각 H2 소제목: 서브 키워드 또는 독자가 검색할 표현 포함
- 마무리: 핵심 요약 + 독자의 다음 행동 (CTA)
- Featured Snippet 狙い 섹션: 해당되면 정의/단계/표 형식으로 구성

출력 형식:
```
[기획자 SEO 설계안]
■ 검색 의도 유형:
■ 독자 시나리오: (검색 상황 1~2문장)
■ 메인 키워드:
■ 서브 키워드:
■ LSI 키워드:
■ 차별화 포인트:
■ Featured Snippet 가능 여부:

■ 제목 후보:
  1. (제목) → SEO 근거: (왜 이 표현이 검색에 유리한가)
  2. (제목) → SEO 근거:
  3. (제목) → SEO 근거:

■ 글 구조:
  ## (H2 섹션명 — 서브 키워드 포함) — (역할 1줄)
  ## ...
■ 예상 읽기 시간:
■ 코드 스니펫 배치:
  - (섹션명): (파일명, 어떤 코드)
```

## STEP 3 — [에이전트 토론] 제목 & 구성 확정

PD + 기획자 + 편집장이 제목과 글 구조에 대해 토론한다.
작가는 이 토론 결과를 기반으로만 집필한다. 토론 없이 바로 집필 금지.

### 토론 방식

각 에이전트가 순서대로 의견을 낸다:

**PD 의견:**
- 핵심 가치 관점에서 제목 후보 평가 (1~3순위 + 이유)
- 글 구조에서 강화하거나 빼야 할 섹션

**기획자 의견:**
- SEO 관점에서 제목 후보 재평가
- "이 키워드로 구글에서 실제로 검색하는 사람이 있는가?" 판단
- 구조에서 검색 유입에 유리한 섹션 배치 제안

**편집장 의견:**
- 독자 관점에서 제목 클릭 여부 판단
- 글 구조가 읽는 흐름에 자연스러운가?
- 3개 후보 중 최종 추천 + 반대 의견 있으면 대안 제시

### 토론 후 확정

3개 에이전트 의견을 종합해 확정한다:
```
[토론 결과]
■ PD: 1순위 (후보 N) — 이유:
■ 기획자: 1순위 (후보 N) — SEO 근거:
■ 편집장: 1순위 (후보 N) — 독자 관점:

■ 확정 제목: (만장일치 또는 다수결, 이견 있으면 대안 제목 추가)
■ 확정 구조:
  ## 섹션명
  ## 섹션명
  ...
■ 작가에게 전달할 추가 지시:
  - (SEO상 반드시 지켜야 할 것)
  - (내용상 반드시 다뤄야 할 것)
```

## STEP 4 — [작가] 초안 집필

토론에서 확정된 제목과 구조를 기반으로 집필한다.
확정 구조를 그대로 따른다. 임의로 섹션을 추가하거나 변경하지 않는다.

**집필 조건:**
- 분량: 3,000자 이상
- 톤: 1인칭, 솔직한 개발자 블로그 톤 (AI 문체 금지)
- 코드 스니펫: 실제 프로젝트 파일에서 인용 (임의 예시 코드 금지)
- 뻔한 서론 절대 금지 ("안녕하세요 오늘은 ~에 대해 알아보겠습니다")

**SEO 집필 조건:**
- 도입부 100자 이내에 메인 키워드 자연스럽게 등장
- H2 소제목마다 서브 키워드 또는 LSI 키워드 1개 이상 포함
- 메인 키워드 본문 내 3~5회 자연스럽게 반복 (keyword stuffing 금지)
- Featured Snippet 狙い 섹션은 정의형("~란 ...이다") 또는 단계형(1. 2. 3.) 형식으로
- 이미지 삽입 위치: [이미지: 설명] 형식으로 표시
- 내부 링크 추천 위치: [내부링크: 관련 주제] 형식으로 표시

출력 형식:
```
[작가 초안]
(전체 본문)
```

## STEP 5 — [편집장] 1차 검토

PD 브리핑 핵심 가치 + SEO 전략 기준으로 검토한다.
칭찬 생략, 문제점과 수정 지시만 출력한다.

**내용 검토 기준:**
- 각 섹션이 PD 브리핑의 핵심 가치와 직결되는가?
- PD가 "다루지 말 것"으로 지정한 내용이 들어가 있는가?
- 핵심 구현 설명이 "왜 이렇게 했는가"까지 다루는가?
- 실제 프로젝트 코드가 인용됐는가? (임의 예시 코드면 지적)
- 도입부가 독자의 문제를 즉시 제시하는가?
- AI가 쓴 것 같은 문체가 있는가? (해당 문장 직접 인용해서 지적)
- 다른 블로그에도 있는 일반론으로 채워진 섹션이 있는가?

**SEO 검토 기준:**
- 도입부 100자 이내에 메인 키워드가 등장하는가?
- H2 소제목에 서브/LSI 키워드가 포함됐는가?
- 메인 키워드가 본문에 3~5회 자연스럽게 반복됐는가?
- Featured Snippet 狙い 섹션이 정의형/단계형 형식인가?
- 분량이 3,000자 이상인가?

출력 형식:
```
[편집장 1차 검토]
🔴 필수 수정 (내용):
-
🔴 필수 수정 (SEO):
-
🟡 권장 개선:
-
```

```

수정 항목:
```

## STEP 7 — [편집장] 최종 결재

수정본을 최종 검토한다. 이 단계를 통과해야 사용자 선택으로 넘어간다.
본문 상태가 "수정 필요"면 작가에게 반려, "승인"이면 STEP 9로 진행한다.

```
[편집장 최종 결재]
■ 본문 상태: 승인 / 수정 필요
  (수정 필요 시 구체적 지시)

■ SEO 최종 점검:
  - 메인 키워드 노출 위치: (도입부 / 소제목 / 본문 N회)
  - 메타 디스크립션 (160자 이내, 클릭 유도):
  - 예상 구글 노출 가능성: 높음 / 중간 / 낮음 + 근거
```

## STEP 9 — 최종본 생성 및 저장

사용자 선택 반영 후 최종본을 저장한다.

파일명 규칙:
- 선택된 제목에서 핵심어 추출 → 영문 슬러그 변환
- 날짜 prefix 포함
- 예: 2026-03-24-claude-code-react-hooks-optimization

저장 경로:
blog-output/
├── 2026-03-24-{slug}.md
└── 2026-03-24-{slug}.html

마크다운 파일 상단 frontmatter:
---
title: (최종 제목)
date: (작성일)
keywords: (메인키워드, 서브키워드1, 서브키워드2, 서브키워드3)
description: (메타 디스크립션 160자 이내)
tags: (티스토리 태그 5개)
---

HTML 파일 조건:

**출력 구조 — 파일 첫 줄은 반드시 `<link ...>`. `<!DOCTYPE>`, `<html>`, `<head>`, `<body>` 절대 금지.**

```
<link href="...Google Fonts...">
<style>...</style>
<div class="bp-wrap">...</div>
<script>...</script>
```

**티스토리 CSS 충돌 방지 원칙:**
- 모든 CSS는 `.bp-wrap` 하위 스코프로 한정 (전역 선택자 금지)
- `position: fixed` 금지 → 토글 버튼은 `position: sticky; float: right`
- 티스토리 자체 CSS에 덮이는 속성은 `!important` 필수:
  - `p`, `h1~h6`, `li`, `strong` → `color !important`
  - `.code-block`, `.code-block pre` → `background !important`
  - `.code-block code`, `p code`, `li code` → `color !important`, `background !important`

**CSS 변수 및 기본 스타일 (.bp-wrap 스코프):**
```css
.bp-wrap {
  --bg: #0d0d0d; --bg2: #161616; --bg-code: #141414;
  --tx: #f0f0f0; --tx2: #a0a0a0; --tm: #606060;
  --ac: #7c6af5; --ac-s: rgba(124,106,245,0.12);
  --bd: #2a2a2a; --cb: #2e2e2e;
  background: var(--bg); color: var(--tx);
  font-family: 'DM Sans', 'Noto Sans KR', sans-serif;
  font-size: 17px; line-height: 1.9;
  max-width: 720px; margin: 0 auto; padding: 2rem 1.5rem;
  animation: bp-fade 0.4s ease-out;
}
.bp-wrap[data-theme="light"] {
  --bg: #fafaf8; --bg2: #f3f3f0; --bg-code: #f0f0ec;
  --tx: #1a1a1a; --tx2: #555550; --tm: #999990;
  --bd: #e0e0d8; --cb: #d8d8d0;
}
```

**폰트:** 제목 `Instrument Serif`, 본문 `DM Sans`, 코드 `JetBrains Mono`
```html
<link href="https://fonts.googleapis.com/css2?family=Instrument+Serif:ital@0;1&family=DM+Sans:wght@300;400;500&family=JetBrains+Mono:wght@400;500&display=swap" rel="stylesheet">
```

```html
<div class="code-block">
  <div class="code-label">bash</div>
  <button class="copy-btn" onclick="this.textContent='복사됨';setTimeout(()=>this.textContent='복사',800);navigator.clipboard.writeText(this.closest('.code-block').querySelector('code').textContent)">복사</button>
  <pre><code>코드 내용</code></pre>
</div>
```

**다크/라이트 토글 — `.bp-wrap` 기준으로 동작:**
```html
<button class="bp-toggle" onclick="const w=this.closest('.bp-wrap');const isLight=w.getAttribute('data-theme')==='light';isLight?w.removeAttribute('data-theme'):w.setAttribute('data-theme','light');this.textContent=isLight?'☾':'☀';localStorage.setItem('bp-theme',isLight?'dark':'light')">☾</button>
```

**초기화 스크립트 (파일 맨 끝):**
```html
<script>
(function(){
  const saved=localStorage.getItem('bp-theme');
  const preferLight=window.matchMedia('(prefers-color-scheme: light)').matches;
  const wrap=document.querySelector('.bp-wrap');
  const btn=document.querySelector('.bp-toggle');
  if(!wrap)return;
  if(saved==='light'||(!saved&&preferLight)){wrap.setAttribute('data-theme','light');if(btn)btn.textContent='☀';}
})();
</script>
```

**절대 금지:**
- `<!DOCTYPE>`, `<html>`, `<head>`, `<body>` 태그
- 전역 선택자 (`p {}`, `h1 {}` 등 `.bp-wrap` 없이 단독 사용)
- `position: fixed`
- `!important` 누락 (`p`, `h1~h6`, `li`, `pre`, `code` 색상/배경)
- Inter, Roboto, Arial, system-ui 폰트
- 그라디언트 배경, `box-shadow` 2개 이상
